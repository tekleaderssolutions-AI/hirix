3.3 Contract & Behavior Rules
Contract Principles
Stable and versioned
Explicit input/output schemas
No business logic
One responsibility per port
Backward compatible evolution
Versioning Strategy
Semantic versioning (MAJOR.MINOR.PATCH)
Non-breaking: MINOR / PATCH
Breaking: MAJOR
All consumers declare version range
Error Handling
All ports throw typed exceptions
No generic exceptions
Retryable vs Non-retryable defined
Timeouts must be configurable
Timeout & Resilience
Timeout is part of contract
Caller decides retry policy
Circuit breaker at adapter level
Not in ports
Data Rules
All requests/responses JSON (or Protobuf)
PII must be encrypted in transport
No sensitive data in logs

Architecture Summary — Key Design Principles
Hexagonal Architecture: all external calls go through Ports → Adapters
Domain Core (L5) has zero dependency on frameworks or infrastructure
Event-driven async backbone (Kafka) decouples all layers
L1 Agents are stateless; only the Orchestrator holds execution state
All I/O are structured, versioned, schema-validated JSON events
LLM calls are minimized: rules-first, LLM only when necessary (cost-aware routing)
Every decision is auditable, traceable, and explainable by design
PII is encrypted at rest & in transit; masked in logs everywhere
Resilience is built-in at every layer: retry, circuit break, saga, DLQ
L7, L8, L9 are horizontal — they apply to every other layer simultaneously
All data access is routed via: L4 (Application) → L3 (Ports) → L2 (Adapters) → L6 (Data & Infrastructure) — ensures consistency, security, and separation of concerns
